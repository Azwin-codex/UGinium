# CHAPTER 15

# Product Language & Microcopy System Specification

---

## Purpose

Language is a critical part of the UGinium experience.

Every word presented to the user should reinforce clarity, professionalism, and confidence.

The objective of Product Language is not to sound futuristic.

It is to communicate naturally while subtly reinforcing the Academy identity.

Users should immediately understand every interaction without learning a new vocabulary.

The interface should speak like a professional product—not a game.

---

# 15.1 Language Philosophy

UGinium communicates with clarity.

Language should be:

Professional

Friendly

Confident

Helpful

Encouraging

Concise

Every sentence should help the user complete a task.

Avoid unnecessary technical jargon.

Avoid theatrical or exaggerated wording.

---

# 15.2 Writing Principles

Every piece of interface text should follow these principles.

Clarity before Creativity.

Brevity before Complexity.

Guidance before Instruction.

Professionalism before Humor.

Consistency before Variation.

Users should never need to interpret interface language.

---

# 15.3 Brand Vocabulary

UGinium introduces only a small number of unique product terms.

These include:

Guild

Mission

Academy Core

Campus

Hall of Legends

XP

Level

All other interface terminology should remain familiar.

Examples:

Profile

Settings

Search

Notifications

Messages

Help

Login

Register

Save

Delete

This balance preserves usability while giving UGinium its own identity.

---

# 15.4 Button Language

Buttons should communicate actions clearly.

Preferred examples:

Join Guild

Create Mission

Edit Profile

Save Changes

Explore Projects

View Portfolio

Invite Members

Publish Project

Continue

Cancel

Avoid vague labels such as:

Click Here

Proceed

Do Action

Submit

Buttons should describe the outcome of the action.

---

# 15.5 Navigation Labels

Navigation should remain familiar.

Examples:

Home

Discover

Guilds

Projects

Portfolio

Messages

Notifications

Settings

Profile

Avoid renaming common navigation items solely for thematic purposes.

---

# 15.6 Empty States

Empty states should motivate action.

Never leave users at a dead end.

Examples:

No Projects Yet

Start building your first project.

No Guilds

Join a Guild or create one to collaborate.

No Portfolio

Add your work to begin showcasing your experience.

No Notifications

You're all caught up.

Every empty state should suggest the next logical step.

---

# 15.7 Success Messages

Success messages should be brief and reassuring.

Examples:

Profile updated successfully.

Mission created.

Guild joined successfully.

Changes saved.

Project published.

Portfolio updated.

Avoid unnecessary celebration.

Confidence comes through simplicity.

---

# 15.8 Error Messages

Every error should explain:

What happened.

Why it happened.

How to fix it.

Poor example:

Something went wrong.

Better example:

We couldn't connect to the server.

Please check your internet connection and try again.

Errors should reduce frustration.

Never increase it.

---

# 15.9 Confirmation Messages

Potentially destructive actions require confirmation.

Examples:

Delete Project

Leave Guild

Remove Member

Archive Mission

Confirmations should clearly describe the consequence of the action.

Avoid vague confirmations.

---

# 15.10 Notifications

Notifications should be informative.

Examples:

You have been invited to join a Guild.

Your project has been approved.

Your profile is now 90% complete.

Three students viewed your portfolio today.

Notifications should communicate useful information rather than drive unnecessary engagement.

---

# 15.11 Onboarding Language

Onboarding should feel welcoming.

Avoid overwhelming users.

Guide students step by step.

Examples:

Complete your profile.

Discover your first Guild.

Explore available projects.

Build your portfolio.

Celebrate progress through encouragement—not pressure.

---

# 15.12 Tooltips

Tooltips explain functionality.

Keep them concise.

Examples:

Guild

Collaborate with students on long-term projects.

XP

Earn experience by contributing to meaningful work.

Portfolio

Showcase your best projects and achievements.

Tooltips should educate without interrupting workflow.

---

# 15.13 Placeholder Text

Placeholder text should provide useful examples.

Good examples:

Search students, projects or Guilds...

Enter your GitHub profile URL

Describe your project...

Avoid generic placeholders like:

Type here...

Enter text...

Placeholder text should guide the user.

---

# 15.14 Tone of Voice

UGinium speaks with confidence.

Never arrogant.

Never overly casual.

Never overly corporate.

The tone should feel:

Supportive

Professional

Modern

Respectful

Encouraging

The platform should always respect the user's intelligence.

---

# 15.15 Accessibility

Language should remain understandable.

Avoid unnecessary abbreviations.

Avoid overly complex vocabulary.

Use clear sentence structures.

Support screen readers through descriptive labels.

Language should be inclusive and accessible.

---

# 15.16 Developer Rules

All interface text should be centralized.

Avoid hardcoded strings.

Support localization.

Maintain consistent terminology across every screen.

Microcopy should use reusable translation keys.

---

# 15.17 Claude Rules

When generating interface text:

Prefer clarity over creativity.

Maintain a consistent tone.

Keep labels concise.

Avoid unnecessary thematic language.

Guide users naturally.

Every message should reduce uncertainty.

---

# 15.18 Review Checklist

Before implementation verify:

□ Is every message easy to understand?

□ Are button labels action-oriented?

□ Are errors helpful?

□ Do empty states encourage action?

□ Does the language remain professional?

□ Is terminology consistent?

□ Can new users understand the interface immediately?

---

# 15.19 Non-Negotiable Rules

Language should reduce cognitive load.

Clarity always wins.

Familiar terminology should be preserved whenever possible.

Academy terminology should only be used where it strengthens the product.

Every word should help the user move forward.

The interface should sound like a trusted professional platform.
