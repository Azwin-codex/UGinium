# CHAPTER 11

# Academy Layout & Spatial System Specification

---

## Purpose

The Layout & Spatial System defines how every interface within UGinium is spatially organized.

It establishes the structural framework that governs positioning, spacing, alignment, responsiveness, and composition.

Unlike traditional web layouts, UGinium layouts should feel engineered rather than assembled.

Every screen should communicate order, precision, and clarity before a user even begins reading.

The objective is to create an architectural rhythm that remains consistent throughout the Academy.

---

# 11.1 Layout Philosophy

Layouts should communicate structure.

Users should instinctively understand where to look next.

Every screen should appear balanced.

Every module should have a logical place.

Whitespace is intentional.

Alignment is deliberate.

Hierarchy is obvious.

Nothing should appear randomly positioned.

---

# 11.2 Academy Shell

Every page exists inside the Academy Shell.

The shell is the permanent operating environment.

It contains:

* Navigation
* Orb Navigation
* Search
* Notifications
* User Controls
* Main Workspace

The shell provides continuity between every page.

Only the workspace changes.

---

# 11.3 Grid System

UGinium follows a responsive grid system.

Desktop

12 Columns

Tablet

8 Columns

Mobile

4 Columns

The grid governs module placement, spacing, and alignment.

Modules may span multiple columns but should always align to the underlying grid.

No layout should ignore the grid system.

---

# 11.4 Layout Hierarchy

Every page follows the same structural order.

Academy Shell

↓

Page Header

↓

Primary Modules

↓

Secondary Modules

↓

Supporting Modules

↓

Footer (if required)

Users should immediately identify the primary purpose of the page.

---

# 11.5 Module Placement

Primary Modules receive the most visual emphasis.

Secondary Modules provide supporting information.

Supporting Modules contain contextual data.

Modules should never compete unnecessarily.

Every module must contribute to the user's current objective.

---

# 11.6 Spatial Rhythm

Spacing establishes hierarchy.

The distance between modules should communicate relationships.

Related information remains visually close.

Unrelated information receives greater separation.

The interface should maintain a consistent visual rhythm across every page.

---

# 11.7 Whitespace Philosophy

Whitespace is an active design element.

Its purpose is to improve:

Readability

Focus

Hierarchy

Navigation

Perceived quality

Never reduce whitespace simply to display more information.

A calm interface improves productivity.

---

# 11.8 Responsive Layout

Responsive behavior should preserve architecture.

Desktop layouts maximize simultaneous information.

Tablet layouts simplify without removing hierarchy.

Mobile layouts stack modules while maintaining their relationships.

Only layout changes.

Architecture remains constant.

---

# 11.9 Alignment Rules

Alignment communicates precision.

Use left alignment as the default.

Align module boundaries consistently.

Maintain predictable spacing.

Avoid arbitrary positioning.

Every edge should feel intentional.

---

# 11.10 Information Flow

Users naturally scan interfaces.

Arrange information accordingly.

Identity

↓

Current Objective

↓

Primary Actions

↓

Supporting Information

↓

Statistics

↓

Utilities

Important information should always appear before supporting details.

---

# 11.11 Visual Balance

Every screen should feel balanced.

Avoid large empty regions on one side with dense content on the other.

Module proportions should create stability.

Visual weight should be distributed naturally.

---

# 11.12 Dashboard Composition

The Academy Core prioritizes:

Student Identity

↓

Progression

↓

Current Missions

↓

Guild Activity

↓

Recommended Opportunities

↓

Signals

↓

Statistics

The dashboard should immediately answer:

"What should I do next?"

---

# 11.13 Layout Constraints

Do not stretch modules unnecessarily.

Do not compress important information.

Maintain comfortable reading widths.

Respect spacing tokens.

Avoid visual clutter.

Every layout should remain breathable.

---

# 11.14 Claude Layout Workflow

Before constructing any page, Claude must:

1. Understand the user's objective.

2. Identify the required modules.

3. Arrange Primary Modules first.

4. Position Secondary Modules.

5. Apply the Academy Grid.

6. Verify responsive behavior.

7. Review visual hierarchy.

Only then should implementation begin.

---

# 11.15 Review Checklist

Before approving a layout verify:

□ Does the page follow the Academy Grid?

□ Are Primary Modules immediately visible?

□ Is spacing consistent?

□ Is whitespace intentional?

□ Does the page feel balanced?

□ Does the hierarchy remain obvious?

□ Is the layout responsive?

□ Does the page feel engineered?

---

# 11.16 Non-Negotiable Rules

Every layout follows the Academy Grid.

Architecture defines placement.

Whitespace improves usability.

Primary Modules always dominate.

Layouts adapt.

They never redesign.

Every page should feel like another room inside the same Academy.
